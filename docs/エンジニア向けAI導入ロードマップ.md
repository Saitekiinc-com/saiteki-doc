# エンジニア向けAI導入ロードマップ：課題解決と未来への適応

## 1. Vision：Saitekiが目指す組織とエンジニア像

AI導入の先に目指すのは、単なる効率化ではありません。Saitekiのエンジニアと組織が以下のような状態になることを目指します。

### 組織の理想像 (Organizational Ideal)
*   **「単純作業ゼロ」の組織**:
    *   定型的なコーディング、テスト記述、ドキュメント更新はAIとシステムが自律的に処理し、人間は「意思決定」と「創造」に100%のリソースを割いている。
*   **「集合知」が即座に使える組織**:
    *   組織内の全ドキュメント、コード、過去の議論がAIによって構造化され、誰でも瞬時に必要な情報にアクセス・活用できる。

### エンジニアの理想像 (Employee Ideal)
*   **AI Orchestrator (AIの指揮者)**:
    *   自らコードを書くだけでなく、AIに適切な指示（プロンプト/コンテキスト）を与え、複数のAIエージェントを指揮して、一人でチーム並みのアウトプットを出す。
*   **Value Creator (価値の創造者)**:
    *   「どう作るか（How）」の悩みから解放され、「何を作るか（What）」「なぜ作るか（Why）」という顧客価値の本質に向き合っている。

## 2. AI駆動開発の成熟度モデル (AI-Driven Development Levels)

| Level | フェーズ名 | 状態定義 | 具体的な業務イメージ |
| :--- | :--- | :--- | :--- |
| **Lv.1** | **Individual** | **個人支援** | **「個人の作業効率化」**<br>コード補完、調査支援、要件定義ドラフト作成など、個人の手元作業をAIで高速化する。 |
| **Lv.2** | **Shared** | **シェア型** | **「合意形成の加速」**<br>HTMLモックの早期生成などで「動くもの」を素早く共有し、ステークホルダーとの合意形成や新しい開発アプローチを実現する。 |
| **Lv.3** | **Autonomous** | **自律型** | **「組織能力の拡張」**<br>オンボーディング計画生成、日報ナレッジ蓄積、エキスパートマッチングなど、AIエージェントが自律的に組織の課題を解決する。 |

#### なぜ3段階なのか？ (Why 3 Levels?)
本ロードマップでは、AI導入の進展を「個人の生産性向上」から「組織全体のプロセス変革」へと至る3つのフェーズで定義しています。
*   **Lv.1 (Individual)**: エンジニア個人の "手" を速くする。
*   **Lv.2 (Shared)**: チーム共有の "モノ" を速く作り、コミュニケーションを変える。
*   **Lv.3 (Autonomous)**: 組織の "仕組み" そのものをAIが自律的に回す。
この段階を踏むことで、現場の混乱を避けつつ、着実に組織能力を高めることができます。

## 3. AIツールの5分類 (AI Tool Categories)

### 3. AIツールの5分類 (AI Tool Categories)

本ロードマップで活用するAIツールは、機能と役割によって大きく3つの大項目、計5つの小項目に分類されます。

| 大項目 (Type) | 小項目 (Category) | 特徴と代表ツール |
| :--- | :--- | :--- |
| **対話型**<br>(Interactive) | **① 支援型AI**<br>(Copilots) | **「副操縦士」**<br>人間の作業を横で支援する。<br>・特徴: コード補完、チャットでの質問応答、エラー解説。<br>・ツール: GitHub Copilot, Cursor, Gemini Code Assist |
| **自律型**<br>(Autonomous) | **② 自律型エージェント**<br>(Agents) | **「自律した働き手」**<br>人間の代わりにタスクを完遂する。<br>・特徴: 計画立案から実行、エラー修正までを自律的に行う。<br>・ツール: Manus, Devin, OpenHands |
| ^ | **③ 一括生成・構築型AI**<br>(Generators) | **「工場」**<br>自然言語から成果物そのものを生成する。<br>・特徴: プロンプトからアプリやUIを一括生成する。<br>・ツール: Bolt.new, v0, Replit Agent |
| **特化型**<br>(Specialized) | **④ ナレッジ・調査支援AI**<br>(Knowledge) | **「賢者」**<br>膨大な情報から回答を導き出す。<br>・特徴: 社内文書やWeb情報を検索・要約し、意思決定を支援する。<br>・ツール: NotebookLM, Perplexity, Glean |
| ^ | **⑤ 特定領域特化型AI**<br>(Specialized) | **「スペシャリスト」**<br>特定の専門業務に特化している。<br>・特徴: セキュリティ診断、テスト生成、ドキュメント作成など。<br>・ツール: Snyk (Security), CodiumAI (Testing), Mintlify (Docs) |

#### なぜこの分類なのか？
AIツールは「チャットができる」という点では似ていますが、**「得意な対話モード」と「生成される成果物」**が決定的に異なるためです。
*   **対話型**: 人間主導。細かい指示で微調整しながら作るのに向く。
*   **自律型**: AI主導。大きな目的を与えて任せるのに向く。
*   **特化型**: 特定タスク（検索、テスト）において汎用モデルを凌駕する性能を持つ。
この分類を理解することで、タスクに応じた「適材適所」のツール選定が可能になります。

## 5. 導入・検証サイクル (The Adoption Cycle)

Lv.2〜3への移行を目指し、以下のサイクルを回します。
**重要なのは、単一プロジェクトでの成功に留めず、複数の案件で得られたAI活用のノウハウを組織全体に蓄積・還流させることです。**

### Step 1: 対象領域の選定 (Selection)
チームへのヒアリングに基づき、AI適用のターゲットを決定します。

#### AI適用判断フレームワーク (Decision Framework)
以下の3つの基準で、そのタスクがAI導入に適しているかを判断します。

1.  **反復性とボリューム (Repetitiveness & Volume)**
    *   「週に何度も発生するか？」「1回あたり15分以上かかるか？」
    *   → **Yes**: 投資対効果が高い。
2.  **プロセスの標準化度 (Standardization)**
    *   「手順書があるか？」「誰がやっても同じ結果になるべきか？」
    *   → **Yes**: AIへの指示（プロンプト）が安定しやすい。
3.  **リスク許容度 (Risk Tolerance)**
    *   「間違いが許されないか？（決済、本番DB操作など）」
    *   → **No**: AIに任せるのは時期尚早、または厳重なHuman-in-the-loopが必要。

#### 選定の視点
*   **視点A: ボトルネックの解消 (Fix Weaknesses)**
    *   「時間がかかっている」「ミスが多い」「やりたくない」作業は何か？
    *   例：レガシーコード調査、単体テスト記述、議事録作成。
*   **視点B: 強みの強化 (Enhance Strengths)**
    *   「既にうまくいっているが、AIでもっと速く・高品質にできる」作業は何か？
    *   例：得意なアーキテクチャパターンのテンプレート化・高速展開、高品質なドキュメントの多言語化。
*   **視点C: 未来への投資 (Future Ready)**
    *   「AIネイティブな新人が入ってきた時に、この作業フローで通用するか？」
    *   例：環境構築のIaC化（コマンド一発で完了するように）、暗黙知のドキュメント化。

### Step 2: 仮説立案とインパクト評価 (Hypothesis & Impact)
*   **選定基準**: 適合性が高く、リスクが許容範囲内であること。
*   **人間によるレビュー**: セキュリティ、ワークフローへの影響、KPI測定可能性を確認。

### Step 3: 小規模検証 (Pilot Verification)
*   パイロットチームで実践し、KPI（時間短縮、品質向上）を測定。
*   **AIネイティブ視点での検証**: 「もしコードが書けない新人がこのツールを使ったら、タスクを完遂できるか？」という観点でも評価する。

### Step 4: ドキュメント化と展開 (Documentation & Expansion)
*   **Docs as Code**: 検証結果とノウハウをMarkdownでGitHub管理。
*   **Cross-Project Sharing (横展開)**:
    *   ある案件で成功したプロンプトや設定を「共通ライブラリ」として他案件にも展開する。
    *   失敗事例も共有し、組織全体の学習コストを下げる。

## 6. ドキュメント管理構成案

将来的に**RAG (Retrieval-Augmented Generation)** での検索拡張を見据え、AIが読み取りやすい構造化されたMarkdownで管理します。

```text
docs/
├── roadmap.md            # 本ロードマップ (Maturity Model & Matrix含む)
├── analysis/             # 現状分析 (Weaknesses & Strengths)
├── experiments/          # 検証記録 (Pilot Log) - RAGの知識源として重要
├── guidelines/           # AI活用ガイドライン
└── knowledge_base/       # 案件横断のプロンプト集・ベストプラクティス
```
