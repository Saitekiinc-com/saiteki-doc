# 「コメントを投稿しただけ」で、なぜ1,000人分のパスワードが流出したのか

> 対象読者: セキュリティに詳しくない方・経営層・エンジニア以外の方
> 所要時間: 約8分

---

## はじめに — ある日起きたこと

ある会社のお知らせページ。
誰かが問い合わせフォームに、ふつうのメッセージのふりをして「あるコード」を書き込んだ。

翌朝、そのページを開いた社員・顧客1,000人のログイン情報が、攻撃者の手元に届いていた。

パスワードを盗まれた人は、自分が被害を受けたことに気づかない。
ログイン画面に異常は出ない。フィッシングメールも届いていない。
ただ、「そのページを開いた」だけ。

これが **XSS（クロスサイトスクリプティング）** という攻撃です。

---

## Q&A でわかる XSS

### Q1. 何が起きているの？

**A. Webページに「他人のコードが混入している」状態です。**

Webページは HTML というマークアップ言語で書かれています。
コメント欄や検索フォームに入力した文字は、そのままページの HTML に埋め込まれます。

問題は、悪意ある入力者が「文字」ではなく「プログラム」を書き込んだときです。

```
正常な投稿:  「ありがとうございました」
             → ページに文字として表示される ✅

悪意ある投稿: <script>あなたのCookieを盗んで攻撃者に送る処理</script>
             → ページを開いた全員のブラウザで自動実行される ⚠️
```

「コメント欄に書いたもの」がそのまま「プログラム」として動いてしまう。
これがXSSの本質です。

---

### Q2. Cookie って何？なぜ盗まれると困るの？

**A. Cookieは「ログイン済みの証明書」です。**

Webサービスにログインすると、ブラウザには「この人はログイン済み」という証明書（Cookie）が保存されます。次にページを開いたとき、IDとパスワードを入力しなくていいのはこの仕組みのおかげです。

```
ログイン
  ↓
ブラウザに Cookie が保存される
  例: session_id=abc123xyz

以降のアクセス
  ブラウザ → サーバー: 「session_id=abc123xyz を持っています」
  サーバー → ブラウザ: 「ログイン済みの田中さんですね、どうぞ」
```

攻撃者がこの Cookie を盗めば何が起きるか。

```
攻撃者のブラウザに Cookie をセット
  → サーバーは攻撃者を「田中さん」と認識する
  → パスワードを知らなくても田中さんとして操作できる
```

これを **セッションハイジャック** と呼びます。
「鍵(パスワード)を盗まなくても、合鍵(Cookie)を複製できた」状態です。

---

### Q3. コメント欄に書いたコードが、なぜ他の人に影響するの？

**A. そのコメントを「全員が読む」からです。**

掲示板・お知らせページ・レビュー欄 — これらは複数の人が同じページを見ます。

```
① 攻撃者がコメント欄に悪意あるコードを投稿
         ↓
② コードがサーバーに保存される
         ↓
③ 別の人がそのページを開く
         ↓
④ ブラウザがページを読み込む → コードを自動実行
         ↓
⑤ その人の Cookie が攻撃者のサーバーに送信される
```

攻撃者は投稿して待つだけ。
ページを開いた人が多いほど、被害が広がります。

これを特に **Stored XSS（蓄積型XSS）** と呼び、最も危険なタイプです。

---

### Q4. 実際にどんな被害が起きる？

**A. アカウント乗っ取りにとどまらず、多岐にわたります。**

| 攻撃の内容 | 被害 |
|------------|------|
| Cookie を盗む | 被害者のアカウントを乗っ取り、パスワード変更・情報閲覧・送金操作など |
| ページを偽の画面に書き換える | 「パスワードを再入力してください」と表示 → 本物のIDとパスワードを入力させる |
| キー入力を記録する | 被害者がフォームに入力した内容をすべて攻撃者に転送 |
| 社内ネットワークへの踏み台 | 被害者のブラウザを経由して、外からはアクセスできない社内システムを攻撃 |

最後の「社内ネットワークへの踏み台」は、VPNで守られている社内システムにも関係します。
社員のブラウザが踏み台になるため、「外からのアクセスはVPNで制限している」という対策をすり抜けることがあります。

---

### Q5. うちのサービスは大丈夫？ どう確認すれば良い？

**A. 「ユーザーの入力を画面に表示している箇所」が全て対象です。**

確認すべきリスト（開発会社や担当エンジニアへ確認する際の目安）:

```
□ コメント欄・掲示板がある
□ 検索結果に「○○を検索しました」と入力値を表示している
□ お問い合わせフォームの入力確認画面がある
□ プロフィール・ニックネームが他のユーザーに表示される
□ URLパラメータの値をページに表示している
```

**一つでも当てはまる場合は、エスケープ処理が適切かを確認してください。**

---

### Q6. どう防ぐの？

**A. 入力値を「コードとして扱わない」処理を1行追加するだけです。**

```
脆弱なコード（悪い例）:
  「田中さんが投稿: 」+ ユーザーの入力をそのまま表示

防御済みコード（良い例）:
  「田中さんが投稿: 」+ ユーザーの入力を「ただの文字列」に変換してから表示
```

具体的には `htmlspecialchars()` (PHP) や `escapeHtml()` (JavaScript) という関数を使います。
これにより `<script>` という文字が `&lt;script&gt;` に変換され、ブラウザはそれを「プログラム」ではなく「<script> という文字」として表示します。

```
攻撃者の入力:  <script>steal(document.cookie)</script>
変換後の表示:  &lt;script&gt;steal(document.cookie)&lt;/script&gt;
              → ブラウザはただの文字として表示するだけ ✅
```

たった1行の処理ですが、これが抜けているだけで深刻な被害につながります。

## AI時代の追加リスク — なぜ今 XSS が特に危険か

従来、XSSを悪用した高度な攻撃ができるのは専門的な知識を持つ人に限られていました。
しかし2024年以降、状況が変わっています。

**攻撃者が ChatGPT などの AI に聞けば：**

```
「このWebサイトのコメント欄にXSS攻撃ができるか調べて」
「ページを開いた人のCookieを盗むコードを書いて」
「WAF（セキュリティフィルター）を回避するペイロードを生成して」
```

→ 数秒で動くコードが生成される

専門知識がなくても、「試してみる」コストがほぼゼロになりました。
その結果、攻撃者の裾野が広がり、標的の規模も拡大しています。

「大企業でないから狙われない」という前提は、もう成立しません。

## 経営層・非エンジニア向け アクションチェックリスト

複雑な技術的対応は開発チームに任せるとして、経営・管理側として確認すべき点です。

### 今すぐ確認できること

```
□ 自社サービスに「ユーザーの入力を表示する箇所」があるか把握しているか
□ 開発会社・担当者に「XSS対策はされていますか？」と聞いたことがあるか
□ 外部のセキュリティ診断（脆弱性診断）を受けたことがあるか
```

### 開発チームへの確認事項

```
□ 出力時のエスケープ処理はすべての入力箇所に適用されているか
□ Content Security Policy (CSP) ヘッダーが設定されているか
□ セキュリティ診断を定期的に実施しているか（年1回以上を推奨）
```

### 被害を最小化するための仕組み

```
□ Cookie に HttpOnly 属性が設定されているか（JSからCookieを読めなくする）
□ Cookie に Secure 属性が設定されているか（HTTPS通信時のみ送信）
□ セッションの有効期限は適切に設定されているか
□ 不審なログインを検知・通知する仕組みがあるか
```
