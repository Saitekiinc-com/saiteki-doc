# ユニットテストの自動生成

## 【背景】なぜこのフローが必要なのか

ユニットテストは品質担保に欠かせない工程ですが、「実装は終わったがテストを書く時間がなかった」という状況は現場でよく起こります。
テストコードの記述は実装と同じくらいの工数がかかることも多く、納期のプレッシャーの中でテストが後回しになりやすい構造的な問題があります。

また、どのケースをテストすべきか（正常系・異常系・境界値）を網羅的に考えること自体にも、一定の経験と時間が必要です。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人の実装 × AIのテストコード生成」** という協働で、テスト作成の工数を削減しながら網羅性を高めます。

1. **テストコードの高速生成**: 実装済みのコードをAIが読み込み、正常系・異常系・境界値のテストケースを一括で生成する。
2. **網羅性の向上**: AIが考慮するケースを一覧化することで、人間が見落としがちなエッジケースを補完する。
3. **テストの保守性確保**: テストのスタイル・命名規則・アサーションの書き方をAIが規約に沿って統一する。

## フローの概要

実装済みのコードとコーディング規約から、ユニットテストコードを自動生成するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・実装済みのコード（テスト対象の関数・クラス）<br>・テストケース一覧（あれば）<br>・使用テストフレームワーク（Jest / Vitest / RSpec など）<br>・コーディング規約・テスト命名規則<br>・モック・スタブの方針 | ・ユニットテストコード（Draft）<br>・テスト対象ケース一覧（正常系・異常系・境界値）<br>・テストのカバレッジ想定<br>・追加でテストすべきケースの提案 |

### 協働フロー

1. 👤 **ユーザー (Human)** — **テスト方針の確定**
    *   テスト対象の関数・クラス・モジュールを決定します。
    *   使用するテストフレームワーク（Jest / Vitest / pytest / RSpec など）とMock方針（spyOn / vi.mock など）を確定します。
    *   テスト命名規則・ファイル配置のルールをAIへの前提として整理します。
    *   *目的: AIが規約に沿ったテストコードを生成できるよう、スタイルの前提を先に固める*

2. 🤖 **生成AI (AI)** — **テストコードの生成**
    *   テスト対象コードを解析し、関数の入力・出力・副作用のパターンを把握します。
    *   正常系（期待通りの動作）・異常系（エラー・例外処理）・境界値（最大値・最小値・空値）のテストケースを生成します。
    *   外部依存（DB・API・ファイルシステム）を持つ場合、モック・スタブを用いたテストを生成します。
    *   テストカバレッジの想定と、さらに追加すべきケースの提案をコメントとして付記します。
    *   成果物：**ユニットテストコード（Draft v0.1）** と **テストケース一覧**

3. 👤 **ユーザー (Human)** — **レビュー・修正・確定**
    *   AIが生成したテストコードを確認し、テストの意図が実装の仕様と一致しているかをレビューします。
    *   モックの範囲が適切か・アサーションが十分か・テストが独立して実行できるかを確認します。
    *   AIが追加提案したケースの中から、優先度が高いものを追加実装します。
    *   テストを実行し、全件グリーンになることを確認します。
    *   成果物：**ユニットテストコード（確定版）**
