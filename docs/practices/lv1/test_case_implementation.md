# 実行用テストケース実装

## 【背景】なぜこのフローが必要なのか

テスト設計書でまとめた「確認すべきこと」を、実際に実行できる形へ落とし込む工程です。
自動化するか手動で行うか、どのツール・レイヤーで実装するかを決定せずにAIへ依頼すると、意図と乖離したコードや手順書が生成されてしまいます。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人の実装方針決定 × AIの変換・生成 × 人のレビュー確定」** という3フェーズの協働で、実行可能・再現可能・自動化可能なテストケースを効率よく作成します。

1.  **実装方針の明確化**: 自動化・手動・使用ツール・実装粒度を人が決定し、AIへの前提条件を固定する。
2.  **AIによる変換・生成**: 確定テストケースを自動テストコードまたは手動テスト手順書へAIが変換する。
3.  **レビューによる品質保証**: AIが生成した実装を人がレビューし、実行可能性・設計意図・保守性を担保する。

## フローの概要

確定済みテスト設計書をもとに、実装可能なテストコードまたは手順書を生成するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **テスト実装前提定義書**<br>確定テストケース一覧（テスト設計書の成果物） | **テストコード（Draft v0.1）**<br>詳細化テストケースドラフト（手動テスト手順書） |

### 協働フロー

1. 👤 **ユーザー (Human)** — **実装方針決定**
    *   テスト実装対象を優先度順に決定します。
    *   各ケースを**自動化するか / 手動で実施するか**を区分します。
    *   使用ツールを決定します（例：JUnit / Playwright / Cypress など）。
    *   実装粒度のルールを確認します（1ケース1テスト関数か、まとめるか など）。
    *   上記をまとめた**「テスト実装前提定義書」**をAIへのインプットとして作成します。
    *   *目的: AIに誤った変換をさせないための実装戦略を確定する工程*

2. 🤖 **生成AI (AI)** — **実行形式への変換**

    **A) 自動テスト → コード生成**
    *   確定テストケースを実行可能なテストコードへ変換します。
    *   テストデータの生成・アサーションの明示・モック定義（必要時）を行います。
    *   エラーパターン対応とリスク紐付けコメントを付与します。
    *   成果物：**テストコード（Draft v0.1）**

    **B) 手動テスト → 手順書整備**
    *   実行手順・確認ポイント・エビデンス取得方法・判定基準・注意点を明確化します。
    *   成果物：**詳細化テストケースドラフト（Draft v0.1）**

3. 👤 **ユーザー (Human)** — **レビュー・確定**
    *   AIが生成したテストコード（自動）または手順書（手動）をレビューします。
    *   **自動テストのレビュー観点**: テスト意図の維持・アサートの妥当性・モックの適切さ・保守性
    *   **手動テストのレビュー観点**: 再現可能か・判定基準が明確か・エビデンスが取れるか
    *   修正・調整を行い、**確定テスト実装（正式版）**として承認します。
