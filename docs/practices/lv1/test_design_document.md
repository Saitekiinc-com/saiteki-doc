# テスト設計書作成

目的
テスト設計書は、要件定義書や仕様書に基づき、どの観点・どの条件で品質を確認するのかを明確にし、テスト実行前の段階で品質保証の妥当性と網羅性を担保するための設計成果物である。単なるテストケースの一覧ではなく、要求事項に対してどのような視点で確認を行い、どこまで検証するのかという品質確認の構造を可視化することを目的とする。
テスト設計書を作成することで、要件とテストの対応関係が明確になり、抜け漏れや重複を事前に防止できる。また、正常系・異常系・境界値・状態遷移などを体系的に整理することで、網羅性を担保しつつ、テストの粒度や観点のばらつきを抑え、属人化を防ぐことができる。さらに、実行前に第三者レビューを可能とし、品質リスクを早期に顕在化させることで、手戻りの削減と実行効率の向上にも寄与する。
すなわち、テスト設計書は「テストを実施するための手順書」ではなく、「品質確認の妥当性を事前に証明するための設計文書」であり、テスト実行フェーズを成功させるための土台となるものである。

## 協働フロー


## ①要求の構造化


### 目的

曖昧な要求を「テスト可能な構造」に変換する

### 人がやること

- 要求の種類を分類する
    - 機能要求
    - 非機能要求
    - 業務要件
    - UI改善
    - 不具合改修
- 要求のスコープを明確にする
    - 対象画面
    - 対象API
    - 対象業務
    - 対象外範囲
- 前提情報を補足する
    - 関連システム
    - 既存仕様
    - 制約条件
    - 過去障害
- AI出力の妥当性判断
    - 粒度が粗すぎないか
    - 余計な想像が入ってないか
    - 抜けている重要要素はないか
- AIにインプットするための想定成果物
① AIインプット用要求整理シート
　→ AIに理解させるための補助層

### AIの作業

- AIにお願いしたいこと
    - 分解
    - 網羅
    - 抽出
    - 拡張
- AIの中間成果物
    - テスト設計対象構造モデル（Draft v0.1）
→ テスト設計の第一成果物（Draft）

## ②リスク特定


### 目的

全部やらないために、「どこが危ないか」を決める

### 人がやること

1. リスク評価基準を定義する
例：
    - 影響度（高 / 中 / 低）
    - 発生確率（高 / 中 / 低）
    - 検出難易度（高 / 中 / 低）
2. ビジネス影響を判断する
    - 売上影響
    - 法令違反
    - 顧客離脱
    - ブランド毀損
3. 最終優先順位を決定する
AIは候補を出す。
**優先度を決めるのは人。**
- AIにインプットするための想定成果物
    - リスク分析前提定義書
→AIにリスクを抽出させるための**分析前提を固定する資料**

### AIの作業

- AIにお願いしたいこと
    - 構造モデルごとにリスク候補を洗い出す
    - 機能リスク・非機能リスクの両面を出す
    - 想定障害を具体化する
    - 過去障害との類似性を示す
    - 根拠を明示する
    - 不足情報を指摘する
- AIの中間成果物
    - リスク候補抽出ドラフト（Draft v0.1）
    →テスト重点対象を判断するための**リスク候補一覧 = 人が優先度を決めるための判断材料**
↓精査(人)
- リスク評価基準を適用
- 影響度を判断
- 発生確率を判断
- 検出難易度を判断
- ビジネス影響を考慮
- 優先順位を確定
中間成果物
- 優先度付きリスク一覧（確定版）

## ③テスト観点の生成


### 目的

優先度の高いリスクに対して、「何を確認すれば、そのリスクを潰せるか」を明確にする。
→リスクをテスト可能な確認視点に変換する工程

### 人がやること

1. 観点生成対象を決める
- 高リスク：詳細展開
- 中リスク：簡易展開
- 低リスク：原則生成しない
👉 全部展開しない
2. 深さ（粒度）を決める
- 機能レベルまで？
- 入力パターンまで？
- 異常系まで？
3. 重複・過剰観点を削減する
AIは広げるのが得意
人は削るのが役割
- AIにインプットするための想定成果物
    - テスト観点生成前提定義書（仮称）
        →リスクに対して、適切な粒度で観点を展開させるための前提資料

### AIの作業

- AIにお願いしたいこと
    - リスクごとに確認観点を展開する
    - 正常系・異常系・境界値を整理する
    - 非機能観点も紐付ける
    - 重複観点をまとめる
    - 「なぜその観点が必要か」を明示する
- AIの中間成果物
    - テスト観点展開ドラフト（Draft v0.1）
    →リスクを潰すための確認ポイント一覧＝テストケース作成の前段階
↓精査(人)
- リスク紐付き確認
- 過剰観点の削減
- 粒度の統一
- 非機能観点の確認
- 品質目標との整合
- 制約との整合
→確定テスト観点一覧

## ④テストケース自動生成


### 目的

確定したテスト観点を、実行可能なテストケースに変換する＝**観点を具体的な確認手順へ落とす工程**

### 人がやること

1. ケース粒度を決める
- 1観点＝1ケース？
- 異常系は分割？
- 境界値はまとめる？
2. 自動化前提かどうか決める
- 手動前提
- APIテスト前提
- E2E自動化前提
3. 過剰ケースを削減する
AIは増やす
人は減らす
4. 実行可能性を確認する
- 環境で再現できるか
- データ準備は可能か
- 手順が現実的か

- AIにインプットするための想定成果物
    テストケース生成前提定義書
    →観点を、適切な粒度・形式でテストケース化させるための前提資料

### AIの作業

- AIにお願いしたいこと
    1. 観点を具体的な手順へ変換
    1. 入力値を具体化
    1. 期待結果を明確化
    1. 前提条件を明示
    1. 自動化適性を示す
    1. リスクとの紐付けを維持
- AIの中間成果物
    - テストケース生成ドラフト（Draft v0.1）
    →実行可能な確認手順を整理するための**初期ケース一覧**＝削減・統合前の素材

## ⑤ケースの削減・統合


### 目的

生成されたテストケースを、
- 重複なく
- 無駄なく
- リスク重点型に整える
＝**最小工数で最大リスクを潰す状態にする工程**

### 人がやること

1.重複ケースの統合
    確認内容
    - 同じ検証目的のケースが複数ないか
    - 入力値だけ違うケースは統合可能か
    - 手順がほぼ同じものはまとめられるか
2.低リスク由来ケースの削減
    確認内容
    - 対応リスクが中・低の場合は削減対象
    - 品質目標未達に直結しないか？
    判断基準：
    このケースがなくても重大障害は防げるか？
3.パターン爆発の抑制
AIは組み合わせを無限に出す。
    確認内容
    - 組み合わせ全探索になっていないか
    - 同値クラスで代表値に置き換えられないか
    - ペアワイズで十分ではないか
4.実行非現実ケースの削除
    - 再現不能
    - 環境依存が強すぎる
    - 実運用で起きない
5.自動化前提の整理
    - 自動化可能なものを優先
    - 手動しかできないが高リスクなら残す
    - 手動×低リスクは削減候補
- AIにインプットするための想定成果物
    - ケース削減判断基準書（仮称）
    →削減判断を属人化させないための基準

### AIの作業

- AIにお願いしたいこと
    - 重複候補抽出
    - 統合可能ケースの提案
    - 同値クラス化の提案
    - 削減影響分析
- AIの中間成果物
    - 削減候補一覧（Draft）
    →人が削減判断するための参考資料
↓精査(人)
- テストケース一覧

## ⑥AIレビュー


### 目的

削減後のテスト設計が、リスクと品質目標を十分カバーしているか検証するための資料

### 人がやること

- 成果物準備

### AIの作業

- AIにお願いしたいこと
    - 高リスク未対応ケースの検出
    - 品質目標未カバー箇所の検出
    - 非機能観点不足の検出
    - 削減によるカバー率低下の検出
    - リスク集中箇所の指摘
- AIの中間成果物
    - 最終抜け漏れ検知レポート（Draft）
    - AI最終レビュー結果
    →人が最終承認するための**抜け漏れ警告レポート**
↓精査(人)
- 修正版テストケース一覧

## ⑦品質可視化


### 目的

テスト設計の結果を、
- 現在の品質状態
- 残存リスク
- リリース可否判断材料
として可視化する。
＝**“やった作業”ではなく“品質状態”を示す工程**

### 人がやること

1. リスクカバー率を整理する
- 高リスク対応率
- 中リスク対応率
- 未対応リスク数
2. 品質目標との整合を示す
- 目標に対する検証状況
- 未達可能性の有無
3. 残存リスクを明示する
- 許容リスク
- 回避できないリスク
- 今後対策予定リスク
4. リリース判断材料を作る
- リリース可能
- 条件付き可能
- リスク高

- AIにインプットするための想定成果物
    - 成果物

### AIの作業

- AIにお願いしたいこと
    - リスクカバー率算出
    - 品質目標との整合整理
    - 残存リスク抽出
    - 状態サマリ作成
    - 上位層向け文章生成
- AIの中間成果物
    - 品質状態サマリ（Draft）
    →現時点の品質状態を一目で理解できる形に整理した資料
↓精査(人：最終承認)
- **確定テストケース一覧**
- 品質状態サマリ
