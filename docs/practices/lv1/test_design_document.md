# テスト設計書作成

## 【背景】なぜこのフローが必要なのか

テスト設計書は「どの観点・どの条件で品質を確認するか」を事前に明確にするための設計成果物です。
単なるケース一覧ではなく、要求とテストの対応関係を構造化することで、抜け漏れ・重複・属人化を防ぎます。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人の判断 × AIの生成・整理」** を反復することで、最小工数で最大のリスクをカバーするテスト設計書を作成します。

1.  **構造化の加速**: 曖昧な要求をAIがテスト可能な構造に変換し、人は妥当性の判断に集中する。
2.  **観点・ケースの網羅**: AIが正常系・異常系・境界値・非機能観点を一括展開し、人が削減・統合で最適化する。
3.  **品質の可視化**: リスクカバー率や残存リスクをAIが整理し、リリース判断を合理的に行えるようにする。

## フローの概要

要求定義書・仕様書から、実行可能なテストケースと品質状態レポートを生成するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **AIインプット用要求整理シート**<br>リスク分析前提定義書・テスト観点生成前提定義書 | **テスト設計対象構造モデル（Draft）**<br>リスク候補・テスト観点・テストケース・抜け漏れレポート・品質状態サマリ |

### 協働フロー

1. 👤 **ユーザー (Human)** + 🤖 **生成AI (AI)** — **要求の構造化**
    *   人が要求の種類（機能 / 非機能 / 業務 / 不具合改修）とスコープを分類します。
    *   前提情報（関連システム・制約・過去障害）を補足し、AIインプット用要求整理シートを作成します。
    *   AIが要求を分解・拡張し、テスト設計対象の構造モデル（Draft）を生成します。

2. 👤 **ユーザー (Human)** + 🤖 **生成AI (AI)** — **リスク特定**
    *   人がリスク評価基準（影響度・発生確率・検出難易度）とビジネス影響を定義します。
    *   AIが機能リスク・非機能リスク・過去障害との類似性をもとにリスク候補一覧を生成します。
    *   人が最終的な優先順位を決定します。*AIは候補を出す役割、優先度は人が判断する。*

3. 🤖 **生成AI (AI)** + 👤 **ユーザー (Human)** — **テスト観点の生成**
    *   AIが優先度の高いリスクに対し、正常系・異常系・境界値・非機能観点を展開します。
    *   人が展開する対象（高リスク：詳細 / 中リスク：簡易 / 低リスク：省略）と粒度を決定します。
    *   人が重複・過剰な観点を削減し、確定テスト観点一覧にまとめます。

4. 🤖 **生成AI (AI)** + 👤 **ユーザー (Human)** — **テストケース自動生成**
    *   AIが確定した観点を具体的な手順・入力値・期待結果へ変換したケースドラフトを生成します。
    *   人がケース粒度・自動化前提・実行可能性を確認し、過剰ケースを削減します。

5. 👤 **ユーザー (Human)** + 🤖 **生成AI (AI)** — **ケースの削減・統合**
    *   AIが重複候補・統合可能ケース・同値クラス化の提案を行います。
    *   人が重複統合・低リスク由来ケースの削減・パターン爆発の抑制を実施します。
    *   *目的: 最小工数で最大リスクを潰せるケース構成に最適化する*

6. 🤖 **生成AI (AI)** + 👤 **ユーザー (Human)** — **AIレビュー**
    *   AIが高リスク未対応・品質目標未カバー・非機能観点不足・カバー率低下を検出します。
    *   人がレポートをもとに最終修正を行い、テストケース一覧を確定させます。

7. 🤖 **生成AI (AI)** + 👤 **ユーザー (Human)** — **品質可視化**
    *   AIがリスクカバー率・品質目標との整合・残存リスク・上位層向けサマリを生成します。
    *   人がリリース可否（可能 / 条件付き / リスク高）を判断し、最終承認します。
