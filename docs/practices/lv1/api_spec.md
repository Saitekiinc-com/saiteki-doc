# API仕様の先行確定

## 【背景】なぜこのフローが必要なのか

フロントエンドとバックエンドが同じチームで並行して開発を進める場合、最大のボトルネックとなるのが**「インターフェースの未確定」**です。

バックエンドのAPIが完成するまでフロントが動けない、API仕様が途中で変わってフロントの実装を作り直す──こうした手戻りを減らすために、開発着手前にAPI仕様（インターフェースの約束事）を早期に確定することが重要です。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人の設計判断 × AIの仕様書生成」** という協働で、openapi.yamlの初稿を高速に作成し、チームで早期に合意形成します。

1. **仕様書生成の高速化**: 要件定義書や画面仕様書をAIが読み込み、エンドポイント・リクエスト・レスポンスの構造を自動で起こす。
2. **漏れの早期発見**: AIが仕様の一覧を展開することで、エラーケース・認証・ページネーションなどの定義漏れを事前に気づける。
3. **モックサーバーへの即時接続**: 確定したopenapi.yamlからモックサーバーを立ち上げ、フロントが即座に開発を開始できる状態にする。

## フローの概要

要件定義書・画面仕様書から、openapi.yamlとモックサーバーを早期に生成するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・要件定義書<br>・画面仕様書<br>・画面一覧・遷移図<br>・機能一覧表<br>・ER図（あれば）<br>・認証方式の決定事項 | ・openapi.yaml（Draft v0.1）<br>・エンドポイント一覧<br>・定義漏れ・曖昧点の指摘リスト |

### 協働フロー

1. 👤 **ユーザー (Human)** — **API設計の前提整理**
    *   対象機能の一覧と、各画面で必要なデータ取得・更新操作を洗い出します。
    *   認証方式（JWT / セッションなど）・ページネーション方針・エラーコードの規則など、設計上の前提を決定します。
    *   AIに渡す資料（要件定義書・画面仕様書・ER図）を準備します。
    *   *目的: AIが適切な仕様書を生成できるよう、判断が必要な前提を先に人が確定する*

2. 🤖 **生成AI (AI)** — **openapi.yaml 初稿生成**
    *   画面仕様書と機能一覧をもとに、必要なAPIエンドポイントを列挙します。
    *   各エンドポイントのリクエストパラメータ・リクエストボディ・レスポンスのスキーマを定義します。
    *   認証が必要なエンドポイントの明示・共通エラーレスポンスの定義も行います。
    *   仕様が曖昧・決定できない箇所はコメントとして指摘します。
    *   成果物：**openapi.yaml（Draft v0.1）** と **定義漏れ・曖昧点リスト**

3. 👤 **ユーザー (Human)** — **レビュー・合意・確定**
    *   AIが生成した初稿をフロントとバックで共同レビューします。
    *   AIが指摘した曖昧点を解消し、設計判断を追記します。
    *   レスポンスのデータ型・Nullableの有無・必須/任意フィールドなど詳細を確定します。
    *   確定後、モックサーバーを立ち上げ、フロントチームへ共有します。
    *   成果物：**openapi.yaml（確定版）** と **モックサーバーURL**
