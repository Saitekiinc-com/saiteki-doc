# コードレビューの質向上

## 【背景】なぜこのフローが必要なのか

コードレビューは品質担保と知識共有の場として非常に重要ですが、現場では以下のような問題が起きがちです。

- **レビューの負担集中**: 経験豊富なメンバーのレビュー待ちがボトルネックになり、開発スピードが落ちる。
- **指摘の粒度のばらつき**: レビュアーによって「てにをは」レベルの指摘から設計レベルの指摘まで粒度がバラバラになる。
- **見落とし**: 大きいPRだとすべてを丁寧に読み切れず、本質的な問題を見逃すことがある。

これらは、AIがレビューコメントの初稿を生成することで大幅に改善できる領域です。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「AIの一次レビュー × 人の最終判断」** という協働で、レビューの質とスピードを同時に高めます。

1. **一次レビューの自動化**: PRのdiffをAIが読んでコメント初稿を生成し、人は本質的な設計判断に集中する。
2. **指摘粒度の標準化**: コーディング規約・レビュー観点をAIへの前提として与え、チームで統一されたレビューコメントを出す。
3. **大型PRの効率化**: 変更量が多いPRでも、AIが変更の全体像をサマリーし、重点確認箇所を絞り込む。

## フローの概要

PRのdiffとコーディング規約から、AIがレビューコメントの初稿を生成し、人が最終判断するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・PRのdiff（変更コード）<br>・コーディング規約<br>・設計書（該当機能の仕様）<br>・レビュー観点リスト<br>・使用フレームワーク・ライブラリ情報 | ・レビューコメント一覧（Draft）<br>・変更のサマリー<br>・重点確認箇所のリスト<br>・潜在的なバグ・パフォーマンス問題の指摘 |

### 協働フロー

1. 👤 **ユーザー (Human)** — **レビュー前提の整理**
    *   レビュー対象のPRとその背景（どの機能・どの設計に基づく実装か）をAIに伝えます。
    *   適用するコーディング規約・レビュー観点（可読性・パフォーマンス・セキュリティなど）を指定します。
    *   *目的: AIが適切な基準でレビューできるよう、プロジェクト固有のコンテキストを与える*

2. 🤖 **生成AI (AI)** — **一次レビューの実施**
    *   PRのdiffを読み込み、変更内容をサマリーします。
    *   コーディング規約・命名規則・コードスタイルの違反を指摘します。
    *   ビジネスロジックの誤り・エッジケースの未考慮・エラーハンドリングの漏れを検出します。
    *   N+1・不要な再レンダリング・重複処理などのパフォーマンス問題を指摘します。
    *   セキュリティ観点（バリデーション・認証・機密情報の取り扱い）のリスクを提示します。
    *   重点的に確認すべき箇所をまとめたリストを出力します。
    *   成果物：**レビューコメント一覧（Draft v0.1）** と **変更サマリー**

3. 👤 **ユーザー (Human)** — **最終レビューと承認**
    *   AIのコメントを確認し、指摘の妥当性を判断します。
    *   AIでは判断できない設計意図・ビジネス要件との整合性・チームの判断基準を加味した最終コメントを追記します。
    *   不要・誤った指摘を除外し、コメントを整理してPRに投稿します。
    *   修正が必要な場合は実装者にフィードバックし、修正後に再レビューを行います。
    *   成果物：**レビュー済みコード（マージ可能な状態）**
