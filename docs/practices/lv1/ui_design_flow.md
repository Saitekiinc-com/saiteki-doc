# 実装前にUIの認識を揃える設計ドキュメントのAI活用

## 【背景】なぜこのフローが必要なのか

開発プロジェクトにおける手戻りの多くは、「エンジニアと企画・デザインの認識ズレ」から生まれます。
画面遷移図やワイヤーフレームはその認識ズレを防ぐための設計ドキュメントですが、  
ゼロから作成するには相応の時間がかかり、後回しにされがちです。

AIを活用することで、要件定義書や仕様の断片的なテキストから、  
構造化された設計ドキュメントのドラフトを素早く生成できます。  
**「じっくり作る前に、まず見せて合意する」** サイクルを個人レベルで実現することが目的です。

### AI活用による課題解決のアプローチ

1. **ドキュメント作成の高速化**: 要件テキストから画面一覧・遷移パターン・コンポーネント定義の骨子をAIが生成し、ゼロから書く手間を省きます。
2. **早期の認識合わせ**: ワイヤーフレームのドラフトをすぐに関係者に見せることで、「思っていた画面と違う」という手戻りを上流で防ぎます。
3. **網羅性の担保**: AIが遷移パターンや共通コンポーネントの抜け漏れをチェックし、人間が見落としがちなケースをカバーします。

## フローの概要

要件定義書や機能一覧をインプットに、画面設計ドキュメント一式をAIと協働で作成するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **機能要件・ユーザーストーリー** | **画面一覧・画面遷移図**（遷移パターン・条件分岐を含む） |
| **画面一覧・デザインシステムの方針** | **共通UIコンポーネント定義**（ボタン・フォーム・ナビゲーション等） |
| **機能要件・画面一覧** | **ワイヤーフレーム**（情報構造・レイアウト・操作の流れ） |

### 協働フロー

1. 👤 **ユーザー** - **要件の整理**
    - 機能要件・ユーザーストーリー・制約条件を洗い出し、AIに渡すテキストを準備します。
    - *ポイント: 「誰が・何をする画面か」が分かるレベルで十分です*

2. 🤖 **生成AI** - **画面一覧と遷移図の生成**
    - 要件テキストを読み込み、画面の一覧と画面間の遷移パターンをMarkdownまたは図（Mermaid等）で生成します。
    - *目的: 作るべき画面の全体像を可視化し、抜け漏れを早期発見するため*

3. 🤖 **生成AI** - **共通UIコンポーネント定義の生成**
    - 画面一覧をもとに、複数画面で共通利用できるUIコンポーネントを洗い出し、定義書を生成します。
    - *目的: 設計の一貫性を保ち、実装工数を削減するため*

4. 👤 **ユーザー** - **内容のレビューと修正**
    - 生成された画面一覧・遷移図・コンポーネント定義を確認し、業務要件との整合性を検証します。
    - 不足・誤りがあれば修正してAIに再生成を依頼します。

5. 🤖 **生成AI** - **ワイヤーフレームの生成**
    - 確定した画面一覧とコンポーネント定義をもとに、各画面のワイヤーフレームを生成します。
    - HTMLでのモック生成や、Markdown・テキストベースのレイアウト定義として出力します。
    - *目的: 関係者が画面のイメージを具体的に確認・議論できるようにするため*

6. 👤 **ユーザー** - **関係者への共有と合意形成**
    - 生成されたワイヤーフレームを関係者に共有し、UIの認識を揃えます。
    - フィードバックをもとにAIと反復的に修正します。
