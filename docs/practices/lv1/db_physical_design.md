# DBの物理実装設計をAIで効率化する

## 【背景】なぜこのフローが必要なのか

ER図や論理設計ができていても、それを実際のDBに実装するための物理設計（テーブル定義・データ型・制約・インデックス）は別の作業です。この工程が丁寧に行われないと、後から「カラムの型が足りない」「制約が抜けていてデータ不整合が起きた」という問題が発生します。

AIを活用することで、論理設計から物理設計への変換をドラフト生成し、
**「実装可能なDB設計を迅速に確定させる」** ことが目的です。

### AI活用による課題解決のアプローチ

1. **物理設計の高速生成**: ER図からテーブル定義・データ型・制約をAIが一括生成し、設計ドキュメントの作成工数を削減します。
2. **制約の網羅**: NOT NULL・UNIQUE・外部キー制約・チェック制約の抜け漏れをAIが防ぎ、データ整合性を設計段階から担保します。
3. **トリガー設計の支援**: 更新日時の自動記録など、業務ルールをDB側で保証するトリガーをAIが提案します。

## フローの概要

論理設計（ER図）をインプットに、実装可能な物理設計ドキュメントを作成するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **ER図・エンティティ定義・業務ルール** | **テーブル定義詳細**（テーブル名・カラム・型・制約・インデックス） |
| **テーブル定義・業務上の整合性要件** | **制約定義**（外部キー・UNIQUE・チェック制約等） |
| **更新ルール・監査要件** | **トリガー定義**（自動更新・ログ記録等） |

### 協働フロー

1. 👤 **ユーザー** - **ER図の確認と物理設計方針の整理**
    - 確定したER図と、データベースの物理設計に関する方針（使用するRDBMS・命名規則・インデックス方針等）を整理します。

2. 🤖 **生成AI** - **テーブル定義書の生成**
    - ER図をもとに、各テーブルのカラム名・データ型・NULL制約・デフォルト値・主キー・外部キーを網羅したテーブル定義書をMarkdown形式で生成します。
    - *目的: 論理設計を実装可能なコードに近い形に落とし込むため*

3. 👤 **ユーザー** - **テーブル定義のレビュー**
    - 生成されたテーブル定義を確認し、データ型の選定・命名の妥当性・ビジネスルールとの整合性を検証します。

4. 🤖 **生成AI** - **制約・インデックス定義の生成**
    - テーブル定義をもとに、データ整合性を保証するための制約（UNIQUE・CHECK・外部キー等）とアクセスパターンに合わせたインデックスを定義します。
    - *目的: アプリケーション側で防ぎきれないデータ不整合をDB側で保証するため*

5. 🤖 **生成AI** - **トリガー定義の生成**
    - 更新日時の自動記録・論理削除フラグの管理など、業務ルール上必要なトリガーを提案・定義します。
    - *目的: 繰り返し発生する更新処理をDB側で一元管理し、アプリケーション側の実装漏れを防ぐため*

6. 👤 **ユーザー** - **最終確認と設計書の確定**
    - テーブル定義・制約・トリガーの全体を確認し、実装チームに共有できる形に整えます。
