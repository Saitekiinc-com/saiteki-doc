# サーバーサイド実装の加速

## 【背景】なぜこのフローが必要なのか

サーバーサイド実装は、APIエンドポイントの定義・ビジネスロジックの実装・DBアクセス・エラーハンドリングなど、複数の層を同時に意識しながら進める作業です。
設計書を読んで「どう実装するか」を考える時間と、コードを実際に書く時間が混在しており、特にロジックが複雑な場合は思考と手作業の切り替えが頻繁に発生します。

また、APIの実装漏れやエラーケースの考慮不足は、結合テストの段階まで発見されにくく、後から修正するコストが大きくなりがちです。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人の設計・ロジック判断 × AIのコード生成」** という協働で、実装速度を上げながら品質を確保します。

1. **エンドポイントの骨格を高速生成**: openapi.yamlをもとにAPIエンドポイントのコードをAIが生成し、人はビジネスロジックの判断に集中する。
2. **ビジネスロジックの変換**: 設計書に記述されたロジックをAIがコードに変換し、人はレビューと例外処理の調整を行う。
3. **実装漏れの防止**: APIエンドポイント一覧・エラーケース・バリデーション要件をAIが列挙し、見落としを事前に検出する。

## フローの概要

サーバーサイドロジック設計書とAPI仕様書から、エンドポイント・ビジネスロジック・DBアクセスのコードを生成するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・openapi.yaml（確定版）<br>・サーバーサイドロジック設計書<br>・ER図・テーブル定義書<br>・コーディング規約<br>・使用フレームワーク・ORMの情報<br>・認証方式・エラーコード規則 | ・APIエンドポイントのコード（Draft）<br>・ビジネスロジックのコード（Draft）<br>・DBアクセス層のコード（Draft）<br>・バリデーション・エラーハンドリングの初稿<br>・未実装・考慮漏れの指摘リスト |

### 協働フロー

1. 👤 **ユーザー (Human)** — **実装前提の整理**
    *   対象APIエンドポイントの範囲を確定します。
    *   使用するフレームワーク（Express / NestJS / Rails / FastAPI など）・ORM（Prisma / TypeORM / Sequelize など）・コーディング規約をAIへの前提として整理します。
    *   認証方式・エラーコードの規則・トランザクション方針など、実装上の設計判断を確定します。
    *   *目的: AIが意図に沿ったコードを生成できるよう、技術的な前提を先に固める*

2. 🤖 **生成AI (AI)** — **コード初稿の生成**
    *   openapi.yamlをもとに、APIエンドポイントのルーティング・コントローラーの骨格を生成します。
    *   サーバーサイドロジック設計書をもとに、ビジネスロジック層のコードを生成します。
    *   ER図・テーブル定義書をもとに、DBアクセス層（Repository / Service）のコードを生成します。
    *   各エンドポイントのバリデーション・エラーハンドリング・レスポンスフォーマットの初稿を出力します。
    *   考慮が必要なエッジケースや認証・認可の漏れを指摘します。
    *   成果物：**APIエンドポイント・ビジネスロジック・DBアクセスのコード（Draft v0.1）**

3. 👤 **ユーザー (Human)** — **レビュー・修正・品質担保**
    *   AIが生成したコードを確認し、ビジネスロジックの正確性・トランザクション整合性・パフォーマンスを判断・修正します。
    *   N+1問題・インデックス未使用・不要なクエリ発行などDBパフォーマンスの観点でレビューします。
    *   セキュリティ観点（SQLインジェクション・認証バイパスの可能性など）を重点的に確認します。
    *   AIが指摘した考慮漏れに対応し、実装を確定します。
    *   成果物：**レビュー済みコード（正式実装）**
