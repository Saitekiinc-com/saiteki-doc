# データの信頼性と性能設計をAIで効率化する

## 【背景】なぜこのフローが必要なのか

データの物理実装（テーブル定義・制約）が完成しても、「データがシステム内をどう流れるか」「大量アクセス時にどう動くか」「障害が起きたときにデータをどう守るか」を設計しないままリリースすると、後から修正が非常に困難になります。

これらは「動いてから考える」ではなく、設計段階でAIと協働して早期に設計しておくべき非機能要件です。
**「後から取り返しがつかなくなる問題を、設計段階で先回りする」** ことが目的です。

### AI活用による課題解決のアプローチ

1. **データフローの可視化**: システム内のデータの動きをAIが整理し、「どこでデータが作られ・どこに伝わるか」を可視化します。
2. **パフォーマンスのボトルネック予測**: AIがアクセスパターンを分析し、インデックス設計・キャッシュ戦略・クエリ最適化の方針を提案します。
3. **バックアップ・リカバリ方針の策定**: 障害シナリオをAIと洗い出し、RPO（目標復旧時点）・RTO（目標復旧時間）に基づいた設計を行います。

## フローの概要

テーブル定義・システムアーキテクチャをインプットに、データの流れと信頼性・性能の設計ドキュメントを作成するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **テーブル定義・機能要件・システム構成** | **データフロー図**（データの生成・更新・削除の流れ） |
| **想定アクセス量・主要クエリパターン** | **パフォーマンス設計**（インデックス・キャッシュ・クエリ最適化方針） |
| **SLA要件・業務上のデータ保全要件** | **バックアップ・リカバリ設計**（RPO/RTO・バックアップ方式・復旧手順） |

### 協働フロー

1. 👤 **ユーザー** - **システム構成と要件の整理**
    - テーブル定義・システムアーキテクチャ・想定アクセス量・データ保全要件を整理します。
    - *ポイント: 「何件のデータが・何回アクセスされるか」の粗い見積もりで十分です*

2. 🤖 **生成AI** - **データフロー図の生成**
    - システム内でデータがどのタイミングで生成・更新・削除されるかを、機能単位でMermaidまたはMarkdown形式で整理します。
    - *目的: データの動きを可視化し、更新の抜け漏れ・整合性の問題を事前に発見するため*

3. 👤 **ユーザー** - **データフロー図のレビュー**
    - 生成されたフロー図を確認し、業務要件と照らして正しいかを検証します。

4. 🤖 **生成AI** - **パフォーマンス設計の生成**
    - 想定アクセス量と主要クエリパターンをもとに、インデックス設計・キャッシュ戦略・クエリ最適化の方針を提案します。
    - *目的: 本番環境で性能問題が起きる前に、設計段階で対策を組み込むため*

5. 🤖 **生成AI** - **バックアップ・リカバリ設計の生成**
    - SLA要件・データ保全の重要度をもとに、バックアップ方式（フル・差分・増分）・保存期間・障害時の復旧手順をドキュメント化します。
    - *目的: 障害発生時に「どのデータをどこまで戻せるか」を事前に設計し、判断コストを下げるため*

6. 👤 **ユーザー** - **設計書の最終確認**
    - データフロー・パフォーマンス・バックアップ設計の全体を確認し、インフラ担当や運用チームと共有できる形に整えます。
