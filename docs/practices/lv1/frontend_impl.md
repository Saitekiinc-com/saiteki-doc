# フロントエンド実装の加速

## 【背景】なぜこのフローが必要なのか

フロントエンド実装は、画面詳細設計・API仕様・状態管理設計など複数の仕様書を同時に参照しながら進める作業です。
「どのAPIを叩けばいいか」「このデータをどのコンポーネントに持たせるか」を設計書を往復しながら判断するだけでも多くの時間を消費します。

また、コンポーネントの骨格・APIの繋ぎ込み・Storeの定義など、ロジックの判断は人が行うが記述量が多い処理は、AIに任せられる余地が大きい領域です。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人の設計判断 × AIのコード生成」** という協働で、実装スピードを高めながら品質を確保します。

1. **コードの骨格を高速に起こす**: 画面仕様書からコンポーネント構造・Storeの初稿をAIが生成し、人はロジックと例外処理の判断に集中する。
2. **API繋ぎ込みのパターン化**: openapi.yamlをもとにAPIクライアントのコードをAIが生成し、繰り返し作業を削減する。
3. **実装漏れの防止**: AIが仕様書を読んで実装対象を一覧化し、見落としを早期に気づける。

## フローの概要

画面詳細設計書とAPI仕様書から、コンポーネント・Store・API繋ぎ込みの初稿を生成するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・画面詳細設計書<br>・openapi.yaml（確定版）<br>・コンポーネント設計書<br>・状態管理（Store）設計<br>・コーディング規約<br>・使用フレームワーク・ライブラリ情報 | ・ページコンポーネントのコード（Draft）<br>・Store定義のコード（Draft）<br>・APIクライアントのコード（Draft）<br>・実装対象一覧と漏れチェックリスト |

### 協働フロー

1. 👤 **ユーザー (Human)** — **実装前提の整理**
    *   対象画面・対象機能の範囲を確定します。
    *   コンポーネントの分割方針・Storeに持たせるデータの範囲を設計します。
    *   使用するフレームワーク（React / Vue / Next.js など）・状態管理ライブラリ（Pinia / Zustand など）・コーディング規約をAIへの前提として整理します。
    *   *目的: AIが意図に沿ったコードを生成できるよう、設計判断を先に固める*

2. 🤖 **生成AI (AI)** — **コード初稿の生成**
    *   画面詳細設計書をもとに、ページコンポーネントの構造を生成します。
    *   openapi.yamlをもとに、APIクライアントのコード（型定義含む）を生成します。
    *   Store設計に基づいて、状態管理の初稿を生成します。
    *   仕様書に記載された画面の要素・バリデーション・エラー表示なども初稿として出力します。
    *   成果物：** コンポーネント・Store・APIクライアントのコード（Draft v0.1）**

3. 👤 **ユーザー (Human)** — **レビュー・修正・品質担保**
    *   AIが生成したコードを確認し、設計意図と一致しているかをレビューします。
    *   ビジネスロジックの正確性・エッジケースへの対応・パフォーマンス上の問題を判断・修正します。
    *   コーディング規約との整合性・命名規則・不要なコードの整理を行います。
    *   状態管理の副作用・メモリリークの可能性など、AIでは判断しにくい箇所を重点的に確認します。
    *   成果物：**レビュー済みコード（正式実装）**
