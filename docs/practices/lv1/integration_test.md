# 結合テストの効率化

## 【背景】なぜこのフローが必要なのか

フロントエンドとバックエンドをそれぞれ単体で動かせても、実際に繋いだときに「レスポンスの形式が想定と違う」「認証トークンの扱いが噛み合っていない」「エラーのステータスコードが統一されていない」といった問題が発生することがよくあります。

こうした結合テストの問題は、単体テストでは検出できず、実際に繋ぎ込んで初めて発覚します。また、テストシナリオを手動で設計・実施するだけでも多くの工数がかかり、後回しにされやすい工程でもあります。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人のシナリオ定義 × AIのテスト設計・コード生成」** という協働で、フロント-バック間の結合品質を効率的に担保します。

1. **テストシナリオの網羅的な展開**: API仕様書と画面仕様書をAIが読み込み、正常系・異常系・境界値のシナリオを一括で列挙する。
2. **テストコードの生成**: 確定したシナリオをもとに、APIリクエスト・レスポンス検証のコードをAIが生成する。
3. **問題の早期発見**: テストシナリオを先に定義することで、インターフェースの不整合を実装前に気づける。

## フローの概要

API仕様書・画面仕様書・実装済みコードから、フロント-バック間の結合テストシナリオとコードを生成するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・openapi.yaml（確定版）<br>・画面仕様書・画面遷移図<br>・実装済みのAPIコード（あれば）<br>・認証方式・セッション管理の仕様<br>・テストフレームワーク情報（Playwright / Cypress / Supertest など） | ・結合テストシナリオ一覧（正常系・異常系）<br>・テストコード（Draft）<br>・インターフェース不整合の指摘リスト<br>・テスト実施優先度の提案 |

### 協働フロー

1. 👤 **ユーザー (Human)** — **テスト範囲・方針の確定**
    *   結合テストの対象APIエンドポイントと画面の範囲を決定します。
    *   使用するテストフレームワーク（Playwright / Cypress / Supertest / k6 など）を確定します。
    *   テスト実行環境（ローカル / ステージング）・テストデータの管理方針を整理します。
    *   *目的: AIが適切なシナリオとコードを生成できるよう、実行環境の前提を先に固める*

2. 🤖 **生成AI (AI)** — **シナリオ展開とコード生成**
    *   openapi.yamlと画面仕様書をもとに、APIとUIの組み合わせで発生しうる操作シナリオを列挙します。
    *   正常系（期待通りの操作フロー）・異常系（エラー・権限不足・タイムアウト）・境界値（大量データ・空値）のシナリオを生成します。
    *   各シナリオのテストコード（リクエスト送信 → レスポンス検証）を生成します。
    *   インターフェースの不整合（フロントの期待するレスポンス形式とAPIの実際のレスポンスのズレなど）を指摘します。
    *   成果物：**結合テストシナリオ一覧** と **テストコード（Draft v0.1）**

3. 👤 **ユーザー (Human)** — **レビュー・実行・修正**
    *   AIが生成したシナリオを確認し、業務上重要なフローが含まれているかレビューします。
    *   テストコードを実行し、失敗したテストの原因（フロント側の問題 / バック側の問題）を特定・修正します。
    *   AIが指摘したインターフェース不整合を解消します。
    *   全シナリオがグリーンになることを確認し、結合テスト完了とします。
    *   成果物：**結合テスト完了レポート**
