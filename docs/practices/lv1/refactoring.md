# リファクタリングの推進

## 【背景】なぜこのフローが必要なのか

開発を進める中で「動いているが読みにくいコード」「重複した処理が散らばっている」「仕様変更のたびに修正箇所が爆発する」という状態は、どのプロジェクトでも徐々に蓄積されていきます。

リファクタリングはこうした技術的負債を解消する重要な工程ですが、「機能開発の優先度に押されて後回しになる」「どこから手をつければいいかわからない」という問題から、実施が後手に回りがちです。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「AIによる問題の可視化 × 人の優先判断 × AIのコード変換」** という協働で、リファクタリングを継続的に実施できる仕組みを作ります。

1. **問題箇所の可視化**: コードをAIが読み込み、重複・複雑度・命名の問題を一覧化することで、着手場所を明確にする。
2. **リファクタリング案の生成**: 問題のある箇所に対してAIが改善案のコードを生成し、人は変更の妥当性を判断する。
3. **ふるまい保持の確認**: リファクタリング前後でテストが変わらずパスすることを確認する手順をAIがサポートする。

## フローの概要

既存コードをAIが分析して問題を可視化し、リファクタリング案を生成するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・リファクタリング対象のコード<br>・コーディング規約<br>・関連するテストコード<br>・リファクタリングの目的・優先観点（可読性 / パフォーマンス / 再利用性など） | ・問題箇所の一覧と改善提案<br>・リファクタリング後のコード（Draft）<br>・ふるまいが変わっていないことの確認ポイント<br>・優先度付きの改善ロードマップ |

### 協働フロー

1. 👤 **ユーザー (Human)** — **リファクタリングの目的・範囲の確定**
    *   対象ファイル・モジュールの範囲を決定します。
    *   優先する観点を明確にします（例：「まず可読性を上げる」「重複を排除する」「パフォーマンスを改善する」）。
    *   関連するテストコードがある場合は、リファクタリング後もパスすることを担保する前提として整理します。
    *   *目的: 何のためにリファクタリングするかを先に決めることで、AIが的外れな提案をしないようにする*

2. 🤖 **生成AI (AI)** — **問題の可視化と改善案の生成**
    *   コードを読み込み、以下の観点で問題箇所を洗い出します：
        *   重複・コピペ箇所の特定
        *   関数・クラスの責務過多（1つの関数が多くのことをやりすぎている）
        *   命名の曖昧さ・一貫性のなさ
        *   ネストの深さ・複雑度の高いロジック
        *   不要なコメント・デッドコード
    *   各問題に対してリファクタリング後のコードを生成します。
    *   変更前後でふるまいが変わっていないことを確認するためのチェックポイントを提示します。
    *   成果物：**問題一覧と改善提案** と **リファクタリング済みコード（Draft v0.1）**

3. 👤 **ユーザー (Human)** — **レビュー・適用・確認**
    *   AIの改善提案を確認し、変更の妥当性・チームの文化・可読性のバランスを判断します。
    *   適用する変更を選択し、コードに反映します。
    *   既存のテストを実行し、全件グリーンになることでふるまいが変わっていないことを確認します。
    *   大きなリファクタリングは段階的に行い、各ステップでコミットして変更を追跡可能にします。
    *   成果物：**リファクタリング済みコード（確定版）**
