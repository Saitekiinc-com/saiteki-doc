# 要件定義の構造化とテストコード生成

## 【背景】なぜこのフローが必要なのか

開発現場において、手戻りが発生する最大の要因は「要件の曖昧さ」と「認識のズレ」です。
従来は、曖昧なビジネス要件がそのまま実装工程に流れ込み、テスト段階で「思っていた動きと違う」という手戻りを生んでいました。

### AI活用による課題解決のアプローチ
本カリキュラムでは、**「AIによる構造化と可視化」**をプロセスの中心に据えます。
AIは単にコードを書くだけでなく、曖昧な日本語のテキストから論理的なマトリクスを生成したり、網羅的なテストケースを洗い出すことに長けています。

1.  **曖昧さの早期発見**: 人間が見落としがちなエッジケースをAIが洗い出す。
2.  **合意形成の加速**: テキストではなく「表（マトリクス）」や「図」を共通言語にして議論する。
3.  **作業の自動化**: 単調なテストコード作成などは100% AIに任せ、人間は「正しさの判断」に集中する。

## フローの概要

要件定義書のテキストから、整合性の確認用マトリクスを作成し、最終的にテストコードまでを導出するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **要件定義書（ドラフト）**<br>ビジネスロジックの複雑な箇所のテキスト | **マトリクス表**（仕様整理用）<br>**単体テスト項目**<br>**テストコード** |

### 協働フロー

1. 👤 **ユーザー (Human)**
    *   ビジネスロジックの要件を確認し、複雑で分かりづらい箇所や、条件分岐が多い箇所を洗い出します。

2. 🤖 **生成AI (AI)**
    *   洗い出された箇所のテキスト定義を読み込み、条件と結果を整理した**「マトリクス表」**を生成します。
    *   *目的: 人間が視覚的に理解・検証しやすくするため*

3. 👤 **ユーザー (Human)**
    *   生成されたマトリクス表を用いて、開発チームと企画チームで認識合わせ（合意形成）を行います。
    *   仕様の抜け漏れや矛盾があれば修正します。

4. 🤖 **生成AI (AI)**
    *   合意されたマトリクス表の論理をもとに、網羅的な**「単体テスト項目一覧」**を生成します。

5. 🤖 **生成AI (AI)**
    *   単体テスト項目をもとに、実際の**「テストコード」**を生成します。
