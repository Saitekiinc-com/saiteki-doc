# データ構造の整合性を確保する設計ドキュメントのAI活用

## 【背景】なぜこのフローが必要なのか

システムのデータ設計は、実装の土台となる重要な工程です。  
ER図やテーブル設計が曖昧なまま実装が進むと、データの整合性が崩れ、後から修正することが非常に困難になります。

しかし現実には、ER図の作成は時間がかかり、関係者への説明コストも高いため、  
設計の議論が十分にできないまま実装が始まるケースが少なくありません。

AIを活用することで、要件テキストからデータ構造のドラフトを素早く生成し、  
**「設計の不整合を実装前に発見する」** サイクルを個人レベルで実現することが目的です。

### AI活用による課題解決のアプローチ

1. **設計ドラフトの高速生成**: 要件定義書からER図・テーブル定義のドラフトをAIが生成し、ゼロから考える手間を省きます。
2. **整合性の早期検証**: CRUD図や関連テーブルの整合性チェックをAIが行い、設計ミスを実装前に発見します。
3. **属人化の防止**: 設計の意図と制約をドキュメント化することで、特定の担当者だけが理解している状況をなくします。

## フローの概要

要件定義書や機能一覧をインプットに、データ設計ドキュメント一式をAIと協働で作成するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **機能要件・エンティティの候補リスト** | **ER図**（エンティティ・属性・リレーション） |
| **ER図・業務上のマスタデータ定義** | **マスタデータ一覧**（コード値・区分値等） |
| **ER図・正規化方針** | **テーブル定義一覧**（テーブル名・カラム・型・制約） |
| **テーブル定義・機能一覧** | **CRUD図**（どの機能がどのテーブルを操作するか） |

### 協働フロー

1. 👤 **ユーザー** - **エンティティの洗い出し**
    - 機能要件・ユーザーストーリーから「システムが扱うデータの種類（エンティティ候補）」を洗い出します。
    - *ポイント: 「〇〇を管理する」「〇〇を記録する」という表現から抽出するとスムーズです*

2. 🤖 **生成AI** - **ER図の生成**
    - エンティティ候補リストと要件テキストを読み込み、エンティティ間のリレーション・多重度・主要属性をまとめたER図をMermaidまたはMarkdown形式で生成します。
    - *目的: データ構造の全体像を可視化し、設計の抜け漏れや矛盾を早期発見するため*

3. 👤 **ユーザー** - **ER図のレビューと修正**
    - 生成されたER図を確認し、業務ルールとの整合性・正規化の妥当性を検証します。
    - 不足・誤りがあれば修正してAIに再生成を依頼します。

4. 🤖 **生成AI** - **テーブル定義一覧の生成**
    - 確定したER図をもとに、テーブル名・カラム・データ型・NOT NULL制約・主キー・外部キーを整理したテーブル定義一覧を生成します。
    - *目的: 論理設計を実装可能な物理設計に落とし込むため*

5. 🤖 **生成AI** - **CRUD図の生成**
    - テーブル定義と機能一覧をもとに、各機能がどのテーブルに対してCreate・Read・Update・Deleteを行うかをマトリクス形式で生成します。
    - *目的: テーブル間の依存関係・更新頻度を把握し、設計の整合性を確認するため*

6. 👤 **ユーザー** - **最終確認と合意形成**
    - テーブル定義・CRUD図を関係者に共有し、データ設計の認識を揃えます。
    - フィードバックをもとにAIと反復的に修正します。
