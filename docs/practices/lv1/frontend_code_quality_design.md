# 保守性を担保するフロントエンドのコード設計をAIで効率化する

## 【背景】なぜこのフローが必要なのか

コードが動くことと、コードが長期的に維持しやすいことは別の話です。テストが書きにくいロジック、重複したステート管理、スタイルの散在——これらは実装時には見えにくく、後から直すと大きなリファクタリングコストになります。

AIを活用することで、純粋関数・カスタムフック・スタイル設計のドラフトを実装前に生成し、
**「保守性の問題を設計段階で防ぐ」** ことが目的です。

### AI活用による課題解決のアプローチ

1. **テスタブルな設計の先行**: 純粋関数として切り出せるビジネスロジックをAIが特定し、副作用のない設計を促進します。
2. **状態管理の整理**: カスタムフックの設計をAIと協働で行うことで、ロジックの重複と複雑な依存を防ぎます。
3. **スタイルの一貫性確保**: デザインシステムの方針をAIがコード設計に落とし込み、属人的なスタイル実装をなくします。

## フローの概要

コンポーネント設計・操作フロー定義をインプットに、品質の高い実装を導く設計ドキュメントを作成するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **フロント側処理フロー・ビジネスロジック一覧** | **純粋関数の設計**（入力・出力・テストケースの骨子） |
| **状態管理が必要な機能・コンポーネント一覧** | **カスタムフック設計**（フック名・管理する状態・公開するAPI） |
| **デザインシステム方針・コンポーネント一覧** | **スタイル設計**（トークン定義・クラス設計・適用方針） |

### 協働フロー

1. 👤 **ユーザー** - **ビジネスロジックの洗い出し**
    - フロントエンドの処理フローから「副作用を持たない計算処理」の候補を洗い出します。
    - *ポイント: 「APIを呼ばず、入力を取って結果を返すだけの処理」が純粋関数の候補です*

2. 🤖 **生成AI** - **純粋関数の設計書生成**
    - ビジネスロジック一覧をもとに、関数名・引数・戻り値・テストケースの骨子をMarkdownで生成します。
    - *目的: テストが書きやすい実装構造を事前に設計するため*

3. 🤖 **生成AI** - **カスタムフック設計書の生成**
    - 複数コンポーネントで共通利用できる状態管理・APIコール・副作用処理を特定し、カスタムフックの設計書を生成します。
    - *目的: 重複したロジックを共通化し、コンポーネントをシンプルに保つため*

4. 👤 **ユーザー** - **設計内容のレビュー**
    - 純粋関数・カスタムフックの設計が、実装の粒度として適切かを確認します。

5. 🤖 **生成AI** - **スタイル設計書の生成**
    - デザインシステムの方針（カラー・スペーシング・タイポグラフィ等）をもとに、CSSカスタムプロパティやTailwindの設定ファイル等、プロジェクトの技術スタックに合ったスタイル設計を生成します。
    - *目的: 見た目の一貫性をコードレベルで保証するため*
