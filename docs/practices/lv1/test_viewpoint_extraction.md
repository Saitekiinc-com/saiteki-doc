# テスト観点抽出（汎用）

## 【背景】なぜこのフローが必要なのか

テスト設計の現場では、観点の抽出が担当者の経験や勘に依存しており、属人化と抜け漏れが慢性的な課題となっています。
プロジェクトが変わるたびにゼロから観点を考え直し、過去の障害で得たはずの知見も次の機能に活かせていない状態が続きます。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人の方針定義 × AIの網羅的展開 × 人の精査・資産化」** という協働フローで、再利用可能なテスト観点カタログを構築します。

1. **属人化の解消**: ドメイン知識と過去障害をAIへのインプットとして構造化し、誰でも同じ水準で観点を引き出せるようにする。
2. **網羅性の向上**: AIが正常系・異常系・境界値・品質特性など多角的な観点を一括展開し、人間の見落としを補う。
3. **ナレッジの資産化**: 精査・整理した観点をカタログとして蓄積し、同種機能や次のプロジェクトで再利用できる状態にする。

## フローの概要

ドメイン特有の観点と過去障害の知見を体系化し、再利用可能なテスト観点カタログを構築するフローです。

| AIへのInput | AIからのOutput |
| :--- | :--- |
| **観点抽出前提定義書**<br>対象ドメイン・レイヤー・抽象度・過去障害の範囲 | **汎用観点展開ドラフト（Draft v0.1）**<br>ドメイン共通観点・正常/異常/境界観点・品質特性別観点・過去障害再発防止観点 |

### 協働フロー

1. 👤 **ユーザー (Human)** — **観点抽出方針定義**
    *   テストの対象ドメイン（例：決済、認証、在庫管理）を決定します。
    *   観点の抽象度（再利用前提か、具体機能前提か）と対象レイヤー（業務 / 機能 / 非機能）を定義します。
    *   参照する過去障害の範囲と、品質特性軸（性能・可用性など）を含めるかを判断します。
    *   これらをまとめた**「観点抽出前提定義書」**をAIへのインプットとして用意します。

2. 🤖 **生成AI (AI)** — **観点展開**
    *   前提定義書をもとに、ドメイン共通観点を洗い出します。
    *   正常系・異常系・境界値の観点を網羅的に展開します。
    *   品質特性（性能・可用性・セキュリティなど）別の観点を追加します。
    *   過去障害の情報から再発防止観点を生成し、想定される抜け漏れも提示します。
    *   *目的: 人間が見落としがちな観点を網羅的に広げ、精査の土台を作るため*

3. 👤 **ユーザー (Human)** — **精査・資産化**
    *   AIが展開した汎用観点ドラフト（Draft v0.1）を精査します。
    *   確認ポイント：抽象度の統一・特定プロジェクト依存の排除・再利用可能か・過剰観点になっていないか・品質特性が偏っていないか。

4. 👤 **ユーザー (Human)** — **再利用テンプレ登録**
    *   精査済みの観点を、ドメイン別・品質特性別・レイヤー別などでカテゴリ分類します。
    *   `#決済` `#二重実行` `#非同期処理` などのタグを付けて検索可能な状態にします。
    *   必須 / 推奨 / 条件付き の再利用ルールと、更新ポリシー（障害発生時・年1回など）を定義します。
    *   *目的: 観点を「資産」として蓄積し、次のプロジェクトで即座に活用できるようにする*
