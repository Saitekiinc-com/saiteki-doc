# 工数見積もりの精緻化

## 【背景】なぜこのフローが必要なのか

開発プロジェクトにおける工数見積もりは、担当者の経験と勘に依存しがちです。
「なんとなく3日かかりそう」という感覚的な見積もりは、スケジュールの遅延や顧客への説明不足につながりやすく、チームの信頼性にも影響します。

また、タスクの見落としも典型的な問題です。実装を始めて初めて「このAPIのエラーハンドリングも必要だった」「テストコードを書く時間が見積もりに入っていなかった」と気づくケースが後を絶ちません。

### AI活用による課題解決のアプローチ

本カリキュラムでは、**「人のドメイン知識 × AIの構造的タスク展開」** という協働で、根拠のある見積もりを高速に作成します。

1. **タスクの構造的分解**: 要件定義書や機能一覧をAIが読み込み、見落としを含む抜け漏れのない作業リストを展開する。
2. **類似タスクとの比較**: 過去の見積もりデータや作業実績をAIに与え、工数感の根拠づけを補助させる。
3. **リスクの可視化**: 不確実性の高いタスクや前提条件の曖昧なタスクをAIが指摘し、バッファを意識的に設ける。

## フローの概要

要件定義書と機能一覧から、タスク一覧と工数見積もりシートを作成するフローです。

| AIにインプットするもの | AIがアウトプットするもの |
| :--- | :--- |
| ・要件定義書<br>・機能一覧表<br>・画面一覧・遷移図<br>・過去の見積もりデータ（あれば）<br>・コーディング規約・技術スタック情報 | ・タスク一覧（実装・テスト・レビューを含む）<br>・タスクごとの工数試算<br>・見積もり根拠メモ<br>・リスク・不確実性の指摘 |

### 協働フロー

1. 👤 **ユーザー (Human)** — **見積もり前提の整理**
    *   対象範囲（どの機能・画面を実装するか）を確定します。
    *   技術スタック・チーム構成・コーディング規約など、AIに与える前提情報を整理します。
    *   過去の類似案件の見積もりデータがあれば参照資料として準備します。
    *   *目的: AIへの指示精度を上げるために、前提を先に固める工程*

2. 🤖 **生成AI (AI)** — **タスク展開と工数試算**
    *   要件定義書・機能一覧をもとに、実装作業をタスクレベルに分解します。
    *   フロントエンド / バックエンド / テスト / レビュー / ドキュメント化 など工程ごとに分類します。
    *   各タスクの工数を試算し、不確実性の高い箇所には仮定条件をメモとして付記します。
    *   成果物：**タスク一覧（Draft v0.1）** と **工数試算シート**

3. 👤 **ユーザー (Human)** — **レビュー・調整・確定**
    *   AIの試算を確認し、チームの実力・環境・依存関係を踏まえて工数を調整します。
    *   見落としているタスクを追加します（AIが見えない社内事情・システム制約など）。
    *   リスクが高いタスクにバッファを追加し、最終的な工数見積もりとして確定します。
    *   成果物：**確定タスク一覧** と **工数見積もりシート（正式版）**
